# Rules: Git / 브랜치 / PR (범용 프로세스 정책)

> 브랜치 네이밍 패턴·흐름은 범용. 단 **기본 통합 브랜치명**(develop/main 등)·**커밋 작성자 표기 정책**은 프로젝트 관례를 조사해 따른다.

## 부모 브랜치 (통합 지점)

- **명시 우선 + 디폴트 = 현재 브랜치**: 사용자가 지정하면 그 브랜치, 미지정이면 현재 체크아웃된 브랜치를 부모로 간주.
- 진입 시 **"부모 브랜치 = X" 한 줄 announce** 후 분기. (추론 금지)
- 부모는 작업 범위에 따라 스토리/에픽/모듈·기능 브랜치 중 하나. 흐름: `feat/* → 부모 → 기본 통합 브랜치(예: develop)`.

## 브랜치 네이밍

- `feat/<모듈>-<단위>` (kebab-case).
- 단위 = **C·D·E 묶음1**(예: `-contract`) / **F 묶음2**(예: `-domain-infra`) / **G는 API마다**(예: `-list-api`, `-create-api`).

## 커밋 (commit-bot)

- 레이어별 분리 커밋(Domain→Infra→Application→Presentation→Module→Docs 순).
- 형식: `<이모지> <레이어>: <요약>` + 본문 bullet (커밋 메시지 언어·이모지 관례는 기존 로그 조사).
- **커밋/PR 작성자 표기 정책은 프로젝트 관례를 따른다** (Co-Authored-By/AI 표기를 금지하는 프로젝트도, 요구하는 프로젝트도 있음 → 기존 로그·설정 조사).
- push/머지는 사람 (단 **G 자동화는 Sonnet이 push + PR까지**, 머지는 절대 금지).

## PR

- **제목**: `<이모지> <기능 요약>` (예: `✨ 게시글 목록 조회 API`).
- **본문** (아래 항목 포함):
  1. **요약** — 무엇을 했는지 1~3줄
  2. **변경 레이어 / 주요 파일**
  3. **관련 티켓 / SPEC** (있으면)
  4. **lint·build 결과** (통과 여부)
  5. **(해당 시) E2E TC 상태** — 단, **E2E 실행 자체는 사람**
  - 작성자 표기는 커밋 정책과 일관(위 참조).
- **머지는 항상 사람.** 충돌은 PR 리뷰 중 사람이 해결.

## G 자동화에서의 PR

- **완료(lint+build 통과)된 API만** Sonnet이 push + PR 생성.
- 실패는 PR 올리지 않음 → 공유문서(`G_PROGRESS.md`)에 `code <워크트리 경로>` + 실패 이유 기록(Opus).
