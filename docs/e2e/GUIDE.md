# E2E 테스트 시나리오 작성 가이드

> 이 문서는 E2E 테스트 스펙 문서의 작성 규칙과 템플릿을 정의합니다.

## 1. 테스트 유형

이 프로젝트는 두 가지 E2E 테스트를 운영합니다.

| 유형 | 목적 | 테스트 파일 | 스펙 문서 |
|------|------|-----------|----------|
| **API Contract Test** | 엔드포인트별 블랙박스 검증 | `test/e2e/*.e2e-spec.ts` | `docs/e2e/p{0-3}-{모듈명}.md` |
| **Scenario Test** | 사용자 여정 흐름 검증 | `test/scenarios/*.scenario.ts` | `docs/e2e/scenario-{이름}.md` |

### API Contract Test

각 API 엔드포인트가 올바르게 동작하는지 독립적으로 검증합니다. 매 TC마다 DB를 클린하여 테스트 간 의존성이 없습니다.

### Scenario Test

실제 사용자의 업무 흐름을 처음부터 끝까지 검증합니다. 여러 API를 체인으로 연결하여 이전 스텝의 결과를 다음 스텝에서 사용합니다.

실행 방법:
- API Contract Test: `npm run test:e2e`
- Scenario Test: `npm run test:scenarios`

## 2. 문서 구조 템플릿

각 시나리오 문서는 아래 구조를 따릅니다.

```markdown
# {모듈명} E2E 테스트 시나리오

> 우선순위: P0 | 최종 업데이트: YYYY-MM-DD
> 관련 컨트롤러: `파일경로1`, `파일경로2`

## 개요

이 모듈의 핵심 비즈니스 흐름을 1~2문장으로 설명합니다.

## 사전 조건 (Preconditions)

테스트 실행 전 준비되어야 할 데이터와 상태를 정의합니다.

| 항목        | 설명                        |
| ----------- | --------------------------- |
| 인증 토큰   | 어떤 역할의 토큰이 필요한지 |
| 선행 데이터 | 부서, 사용자 등             |
| 외부 서비스 | Mock 필요 여부              |

## 테스트 시나리오

### TC-{모듈코드}-{번호}: {시나리오 제목}

> **분류**: Happy Path | Error Case | 권한 검증
> **대상 API**: `{HTTP 메서드} {경로}`
> **테스트 상태**: [ ] 미작성 | [ ] 작성완료 | [ ] 검증완료

**Given** (사전 상태)

- {조건 1}
- {조건 2}

**When** (행동)

- {API 요청 설명}

**Then** (기대 결과)

- 응답 상태: `{HTTP 상태 코드}`
- 응답 본문: {검증할 필드/구조}
- 부수 효과: {DB 변경, 이벤트 발행 등}

---
```

## 3. E2E 시나리오 선정 원칙

E2E 테스트의 목적은 **"전체 흐름이 연결되어 잘 동작하는가"**를 확인하는 것입니다.
단위 수준의 검증(입력 검증, 경계값 등)은 유닛 테스트로 커버하고, E2E에서는 제외합니다.

### E2E에 포함할 것

| 분류                   | 필수 여부    | API당 개수          | 설명                                      |
| ---------------------- | ------------ | ------------------- | ----------------------------------------- |
| **Happy Path**         | 필수         | 1~2개               | 정상 동작 확인                            |
| **인증/인가 검증**     | 필수         | 컨트롤러당 대표 1개 | 401/403 확인 (같은 컨트롤러 내 중복 금지) |
| **핵심 비즈니스 규칙** | 해당 시 필수 | 규칙당 1개          | 상태 전이 위반, 소유권, 중복 방지 등      |

### E2E에서 제외할 것 (유닛 테스트 영역)

| 분류               | 예시                                                                      | 테스트 위치                                   |
| ------------------ | ------------------------------------------------------------------------- | --------------------------------------------- |
| **입력 검증**      | 필드 누락, 타입 오류, 길이 초과, 형식 검증 (`@IsString`, `@MaxLength` 등) | DTO/Pipe 유닛 테스트                          |
| **Edge Case**      | 빈 문자열, 매우 긴 문자열, 공백, 잘못된 UUID 형식, Body 미전송            | DTO/Pipe 유닛 테스트                          |
| **중복 인증 검증** | 같은 역할 제한을 가진 같은 컨트롤러 내 API들의 반복 401/403 검증          | 대표 1개만 E2E, 나머지는 미들웨어 유닛 테스트 |

## 4. 시나리오 분류 기준

E2E에 포함되는 시나리오를 아래 순서로 작성합니다.

### 4.1 Happy Path (정상 시나리오) - 필수

> 해당 API의 가장 기본적인 성공 케이스

- 유효한 인증 토큰 + 올바른 요청 데이터
- 정상 응답 (200, 201 등) 확인
- 응답 데이터 구조 검증

### 4.2 인증/인가 검증 - 필수

> 역할 기반 접근 제어가 올바르게 동작하는지 확인

| 시나리오                            | 기대 결과          |
| ----------------------------------- | ------------------ |
| 토큰 없이 인증 필수 API 요청        | `401 Unauthorized` |
| 일반 관리자로 최고 관리자 전용 요청 | `403 Forbidden`    |
| 사용자 토큰으로 관리자 API 요청     | `401 Unauthorized` |

> **주의**: 같은 컨트롤러 내에서 동일한 역할 제한을 가진 API가 여러 개일 경우, **대표 API 1개만** 인증/인가 시나리오를 작성합니다.

### 4.3 핵심 비즈니스 규칙 검증 - 해당 시 필수

> 도메인 규칙에 의한 거부 케이스

- 부서 타입 규칙 위반 (예: CENTER 부서 중복 생성, 비 CENTER 부서에 parentId 누락)
- 삭제 제한 (예: 소속 구성원이 있는 부서 삭제 시도)
- 중복 데이터 생성 방지 (예: 중복 부서명)
- 기대 결과: `400`, `404`, `409` 등 도메인별 에러 코드

## 6. 역할 정의

시나리오에서 사용하는 역할을 통일합니다.

### 관리자 계열

| 역할        | 코드         | 설명                                 |
| ----------- | ------------ | ------------------------------------ |
| 최고 관리자 | `SuperAdmin` | 플랫폼 전체 관리 (부서/직위 CRUD 등) |
| 일반 관리자 | `Admin`      | 회원/공지사항 조회 등 기본 관리      |

### 사용자 계열

| 역할          | 코드        | 설명                            |
| ------------- | ----------- | ------------------------------- |
| 인증 사용자   | `User`      | 소셜 로그인한 일반 사용자       |
| 비인증 사용자 | `Anonymous` | 로그인하지 않은 상태 (공개 API) |

> **참고**: 이 프로젝트는 관리자 인증(`/v1/admin-auth`)과 사용자 인증(`/v1/user-auth`)이 분리되어 있습니다. 관리자 토큰과 사용자 토큰은 서로 호환되지 않습니다.

## 7. 작성 예시

아래는 관리자 인증 모듈의 시나리오 예시입니다.

### TC-AAUTH-001: 유효한 자격증명으로 관리자 로그인 성공

> **분류**: Happy Path
> **대상 API**: `POST /v1/admin-auth/login`
> **테스트 상태**: [ ] 미작성

**Given**

- 등록된 관리자 계정이 존재함 (loginId: `admin`, role: `SUPER_ADMIN`)
- 올바른 비밀번호가 설정되어 있음

**When**

- `POST /v1/admin-auth/login` 요청
- Body: `{ "loginId": "admin", "password": "valid-password" }`

**Then**

- 응답 상태: `200 OK`
- 응답 본문에 `accessToken`, `refreshToken` 포함
- 응답 본문에 관리자 기본 정보 포함

---

### TC-AAUTH-002: 잘못된 비밀번호로 관리자 로그인 실패

> **분류**: Error Case
> **대상 API**: `POST /v1/admin-auth/login`
> **테스트 상태**: [ ] 미작성

**Given**

- 등록된 관리자 계정이 존재함

**When**

- `POST /v1/admin-auth/login` 요청
- Body: `{ "loginId": "admin", "password": "wrong-password" }`

**Then**

- 응답 상태: `401 Unauthorized`
- 에러 코드: `INVALID_CREDENTIALS`

## 8. Then 검증 원칙 (어설션 가이드라인)

E2E 테스트는 **클라이언트 관점에서 API 계약을 검증**합니다. 내부 구현 세부사항은 유닛 테스트 영역입니다.

### 8.1 검증해야 하는 것 (API 계약)

| 대상                    | 예시                                       | 이유                        |
| ----------------------- | ------------------------------------------ | --------------------------- |
| **응답 상태 코드**      | `200`, `401`, `403`                        | API 계약의 가장 기본        |
| **응답 필드 존재 여부** | `accessToken`, `refreshToken`이 존재하는가 | 클라이언트가 의존하는 필드  |
| **에러 코드**           | `errorCode: 'INVALID_CREDENTIALS'`         | 클라이언트의 분기 처리 기준 |

### 8.2 검증하지 말아야 하는 것 (구현 세부사항)

| 대상                               | 예시                                       | 대안                                            |
| ---------------------------------- | ------------------------------------------ | ----------------------------------------------- |
| **토큰 내부 형식**                 | JWT `.` split 3파트, refreshToken `:` 파싱 | 발급된 토큰으로 보호된 API를 호출하여 간접 검증 |
| **응답 값 파싱 후 내부 구조 대조** | tokenId 추출 후 DB 레코드와 매칭           | 불필요 — 다음 API 호출 성공으로 검증            |
| **입력 검증 에러**                 | 필드 누락, 타입 오류, 길이 초과            | DTO/Pipe 유닛 테스트                            |

### 8.3 DB 부수효과 검증 기준

DB 직접 조회는 **아래 두 조건을 모두 만족할 때만** 허용합니다.

1. **API 응답으로 확인할 수 없는 값**인가?
2. **비즈니스/보안 요구사항의 핵심**인가?

| 예시                                  | 허용 여부 | 이유                                      |
| ------------------------------------- | --------- | ----------------------------------------- |
| Token Rotation 시 이전 토큰 삭제 확인 | 허용      | 보안 핵심, API 응답에 없음                |
| `lastLoginAt` 갱신 확인               | 허용      | 감사 요구사항, API 응답에 없음            |
| 새 refreshToken DB 저장 확인          | 불필요    | 다음 refresh 호출 성공으로 간접 검증 가능 |

### 8.4 토큰 유효성 검증 방법

토큰이 올바른지는 **형식을 들여다보는 것이 아니라, 그 토큰을 사용해보는 것**으로 검증합니다.

```typescript
// Bad: 구현 세부사항 검증
expect(body.accessToken.split('.')).toHaveLength(3);

// Good: 동작 검증 — 발급된 토큰으로 보호된 API 호출
const profileResponse = await request(app.getHttpServer())
  .get('/api/v1/admin/admins')
  .set('Authorization', `Bearer ${body.accessToken}`);
expect(profileResponse.status).toBe(200);
```

> **참고**: Martin Fowler — [The Practical Test Pyramid](https://martinfowler.com/articles/practical-test-pyramid.html), Kent C. Dodds — [Write Tests. Not Too Many. Mostly Integration.](https://kentcdodds.com/blog/write-tests)

## 9. 작성 시 주의사항

1. **한 시나리오 = 한 가지 검증 목적**: 여러 검증을 하나에 섞지 않는다
2. **구체적인 값 사용**: "유효한 데이터" 대신 실제 예시 값을 작성한다
3. **부수 효과 명시**: DB 변경, 이벤트 발행 등 부수 효과를 기술한다 (8.3 기준 충족 시)
4. **실제 코드 참조**: 시나리오 작성 시 컨트롤러/DTO 코드를 확인하여 정확한 필드명과 검증 규칙을 반영한다
5. **인증 체계 구분**: 관리자 인증(`@AdminAuth`, `@RequireSuperAdmin`)과 사용자 인증(`@UserAuth`)을 명확히 구분한다
6. **테스트 상태 업데이트**: 테스트 코드 작성 후 상태를 갱신한다
